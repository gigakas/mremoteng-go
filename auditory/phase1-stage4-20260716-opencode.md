## 1. Code quality

- `Deserialize` centralizes root metadata, version validation, authentication and full-file handling (`internal/serialize/xml/deserialize.go:68`). Errors are typed and wrapped with context.
- Version mapping is cumulative: the 2.6 baseline and inheritance flags (`node26.go:5`, `node26.go:92`), then 2.7 additions (`node27.go:5`), then 2.8 disk/environment changes (`node28.go:5`). This avoids three divergent full copies.
- Recursive tree construction preserves order, IDs and container expansion (`node.go:17`). Missing IDs use stage 1.1's UUID constructor.
- Attribute helpers reproduce C# permissive defaults and ignore qualified attributes, preventing prefixed metadata from overriding exact unqualified attributes.
- Secrets are authenticated before use; root, full-file and node ciphertext whitespace is normalized to C# `Convert.FromBase64String` behavior (`deserialize.go:166`, `node.go:76`).
- Nine tests cover nested v28 data/inheritance/secrets, full-file encryption, KDF 5000, version-specific drives, malformed defaults, typed errors, wrong passwords, namespaced collisions and wrapped Base64 (`deserialize_test.go:13-177`). Three `Protected` values are independent vectors generated by the C# app.

## 2. Performance

- XML is decoded once into a compact recursive representation, then walked once: O(elements + attributes). No reflection is used in production.
- PBKDF2/GCM work is required by file format. Each individually encrypted credential derives its own key, matching the C# implementation.
- Full-file mode temporarily holds decrypted inner XML and its parsed nodes; connection files are user configuration data and bounded well below protocol payload sizes. Streaming would complicate authentication without practical benefit.

## 3. Architecture

- Stage code is confined to `internal/serialize/xml/` and imports `connection`/`security` read-only.
- A prerequisite, separately committed change (`0cc9a40`), added configurable PBKDF2 iterations to `internal/security`; this was required by valid C# files and the official 5000-iteration fixture.
- Only AES/GCM is accepted, matching Phase 1's supported modern provider. Unsupported Twofish/Serpent or CCM/EAX metadata returns `ErrUnsupportedCipher` rather than silently selecting another algorithm.
- Node credential corruption is intentionally fail-fast instead of C# partial-node recovery. Returning a partially trusted tree without a warning channel would hide credential loss; errors identify the affected node.
- Pre-2.6 legacy whole-file wrapping is outside the explicit v26/v27/v28 stage. Legacy AES-CBC remains available in `internal/security` for later corpus/import integration.
